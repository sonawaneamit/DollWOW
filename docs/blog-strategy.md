## Recommendation

Yes: DollWow should add SEO blog posts, but they should be treated as an evergreen **Learning Center**, not a chronological blog. The money pages and collection pages will target commercial searches. The blog/learning center should target long-tail questions, comparison searches, buyer objections, care topics, customization explanations, privacy concerns, and scam-prevention searches.

This matters because DollWow is a new domain. It will be hard to quickly beat entrenched competitors for `sex doll` and `sex dolls`, but it can start earning:

- Long-tail informational traffic.
- Commercial investigation traffic.
- Internal links into collection and product pages.
- AI Overview / answer-engine visibility.
- Trust signals for nervous buyers.
- Content assets that support paid, email, support, and sales follow-up too.

## Blog Vs Guide Vs Collection

Use this rule:

| Search Intent | Best Page Type |
| --- | --- |
| “buy / shop / for sale / brand / material / size” | Collection or product-listing page |
| “best / review / compare / vs / price” | Guide, roundup, or comparison page |
| “how / what / can / is / care / storage / laws” | Learning Center article |
| “specific product model” | PDP |
| “competitor alternative” | Comparison/alternative page |

Do not create separate thin blog posts for every 10-volume question. Fold small questions into FAQ sections on stronger pages.

## Shopify And Headless Notes

The blog itself should live in the headless app, not Shopify blog, unless the team explicitly chooses Shopify as a CMS. Since DollWow is headless and already has repo-managed content, preferred options are:

1. Markdown/MDX files in the repo under `content/learn/`.
2. A lightweight content collection registry in TypeScript.
3. Shopify metaobjects only if the team wants non-developer editing in Shopify Admin later.

If using Shopify metaobjects:

- Create a `learning_article` metaobject type.
- Fields: title, slug, primary_keyword, secondary_keywords, excerpt, body, category, status, last_reviewed_at, reviewer, faq_json, related_collection_handles.
- Only render `status=approved` on the frontend.

For now, repo-based Markdown/MDX is faster and safer.
